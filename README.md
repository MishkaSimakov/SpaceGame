# Веб-версия игры "Космические баталии"

## Контейнеры

Проект можно удобно запустить, используя docker compose. Для этого достаточно выполнить:

```shell
docker compose up
```

Структура контейнеров:

1. База данных. Используется postgres, который запускается в отдельном контейнере. Остальные контейнеры могут
   взаимодействовать с ним по сети.
2. Сервер. Доступен на порту 3000.
3. adminer. Доступен на порту 8080. Через него можно удобно администрировать базу данных.

### Тестовые пользователи

При запуске сервер создаёт двух тестовых пользователей (`DatabaseManager.fakeUsers()`). У каждого пароль совпадает с
логином:

- логин `first` / пароль `first`
- логин `second` / пароль `second`

Ими можно войти на `http://localhost:3000/login`. Для игры нужны два игрока, поэтому доступны оба аккаунта.

## Сервер

Всё держится на действиях (Action). С помощью них передаются сообщения между различными компонентами сервера. Этих
компонентов несколько:

1. saga - отвечает за оркестрацию всего процесса. Именно в ней заложены правила игры и вся основная логика. При этом
   сама saga напрямую никогда не меняет состояния. Она лишь испускает действия вида "ЧтоТоНадоСделать".
2. reducer - отвечает за изменение состояния игры (GameState). Они вызываются из saga через действия.
3. logger - записывает все действия и изменения состояния в файл
4. client - отвечает на запросы от saga

При client может находиться физически на другой машине. Для этого сделан механизм передачи действий через сокеты.
Действия, названия которых имеют вид: `smthRequest` и `smthResponse` (где smth - произвольная строка) передаются от
машины сервера на машину клиента и наоборот через сокеты. Для работы этого механизма у таких действий должен быть
параметр `player` внутри объекта `payload`. В этом параметре должен содержаться id игрока, которому нужно передать это
действие.

## API

### POST `/api/v1/register`

**Описание**: Создаёт нового пользователя

**Запрос**:

```json
{
  "name": "Misha",
  "password": "abracadabra"
}
```

**Успешный ответ**:

```json
{
  "token": "alkjdsiofo2u9843217491fds"
}
```

**Ошибка**:

```json
{
  "error": "user already exists"
}
```

### POST `/api/v1/login`

**Описание**: Выполняет вход в аккаунт пользователя. Вход осуществляется за счёт выдачи токена.

**Запрос**:

```json
{
  "name": "Misha",
  "password": "abracadabra"
}
```

**Успешный ответ**:

```json
{
  "token": "alkjdsiofo2u9843217491fds"
}
```

### POST `/api/v1/users/login`

**Описание**: Выполняет вход в аккаунт пользователя. Вход осуществляется за счёт выдачи токена.

**Запрос**:

```json
{
  "name": "Misha",
  "password": "abracadabra"
}
```

**Успешный ответ**:

Файл логов.

## Валидаторы

Ответы пользователя нужно валидировать. На данный момент для обеспечения безопасности используется идея валидации на
входе, то есть проверка данных происходит только на их входе в сервер, после чего предполагается, что всё корректно.
Следствием является то, что значениям в полях `request` можно доверять. Например, если `request.payload` содержит id
игрока, то можно считать, что он корректный.
